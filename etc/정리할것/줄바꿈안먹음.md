## CommitMessage
- ❌`\n`
  ```
  "A\n - B"
  ```
- ✅진짜줄바꿈
  ```
  "A
  - B"
  ```


##
진짜 Enter의 정체 (LF vs CRLF)
운영체제(OS)마다 "엔터"를 정의하는 방식이 조금씩 다릅니다.

리눅스/맥 (LF): Line Feed라고 부르며, 16진수로는 0x0A, 10진수로는 10번이라는 값을 가집니다. (이게 우리가 흔히 말하는 \n의 실체입니다.)

윈도우 (CRLF): Carriage Return(0x0D) + Line Feed(0x0A) 두 개를 연달아 씁니다. 옛날 타자기에서 줄을 바꾸고 종이를 맨 앞으로 밀던 동작에서 유래했죠.



##
왜 \n 텍스트는 안 되고, 엔터는 될까?
이 차이를 이해하면 질문자님의 궁금증이 완벽히 해결될 거예요.

① git commit -m "feat: \n details" (텍스트 입력)
입력값: \(92번) + n(110번)

컴퓨터가 보는 것: "아, 92번이랑 110번 글자를 연달아 저장해달라는 거구나."

결과: 화면에 그대로 \n이 보임.

② git commit -m "feat: [진짜 엔터] details" (엔터 입력)
입력값: 0x0A (10번 바이트)

컴퓨터가 보는 것: "아, 10번 바이트가 들어왔네? 이건 줄을 바꾸라는 명령어니까 여기서 데이터를 끊고 다음 줄로 넘겨야지."

결과: 데이터 자체가 물리적으로 줄이 나뉜 상태로 저장됨.



##
요약하자면
\n은 우리가 컴퓨터에게 **"야, 나중에 줄바꿈(10번 바이트)으로 해석해줘"**라고 부탁하는 대체 기호일 뿐이고, 진짜 엔터는 그 즉시 10번 바이트(LF)라는 실제 데이터를 꽂아버리는 행위입니다.

터미널 환경에서는 "부탁(기호 해석)"을 기본적으로 거절하기 때문에, "실제 데이터(엔터)"를 직접 때려 넣어야만 줄바꿈이 일어나는 것이죠!



## `\n` vs. `Enter`
`\n`
`\n`.예약어에불과할뿐 진짜엔터ㄴㄴ.
\n은 **"줄바꿈을 하고 싶을 때 사용하는 약속된 기호(Escape Sequence)"**일 뿐이지, 그 자체가 컴퓨터가 인식하는 **"데이터로서의 줄바꿈"**은 아닙니다.
\n은 "번역"이 필요한 문자입니다
우리가 코드나 터미널에 \n이라고 쓰는 건, 컴퓨터에게 일종의 주문서를 던지는 것과 같습니다.
주문: "야, 나중에 이 글자를 읽을 때 **진짜 줄바꿈(ASCII 10번)**으로 바꿔서 처리해줘."
문제: 하지만 git commit -m 같은 일반적인 터미널 명령어는 이 주문서를 해석(번역)해주는 기능이 꺼져 있습니다. 그래서 그냥 "역슬래시랑 n이네?" 하고 글자 그대로 저장해버리는 거죠.


진짜 엔터는 "날것(Raw)의 데이터"입니다
반면, 키보드로 직접 엔터를 치는 행위는 주문서를 쓰는 게 아니라 **결과물(줄바꿈 신호)**을 직접 꽂아넣는 것입니다.
동작: 엔터를 누르는 순간, 보이지 않는 **제어 문자(LF, Line Feed)**가 메모리에 즉시 기록됩니다.
결과: 번역 과정이 필요 없으니 Git도 "오, 줄이 바뀌었네?" 하고 그대로 받아들입니다.


##
- "진짜 엔터"는 데이터 자체가 줄바꿈인 것이고, \n은 줄바꿈을 해달라는 약속된 기호일 뿐입니다. 터미널은 기본적으로 이 기호를 해석해주지 않기 때문에 발생하는 현상이에요.

##
충분히 당황하실 수 있는 부분이에요! 보통 자바(`System.out.println`)나 C언어 같은 프로그래밍 언어에서는 `\n`이 당연히 줄바꿈으로 작동하니까요.

하지만 **터미널(셸, Shell)** 환경에서는 이야기가 조금 다릅니다. 왜 안 먹히는지 이유를 아주 쉽게 짚어 드릴게요.

---

## 1. 셸(Shell)은 `\n`을 "그냥 글자"로 봅니다
터미널에서 명령어를 입력할 때, 셸은 기본적으로 따옴표 안에 있는 내용을 **일반 텍스트(Literal String)**로 취급합니다.

* **프로그래밍 언어:** 컴파일러나 인터프리터가 코드를 읽으면서 `\`를 만나면 "아, 이건 특수 기호구나!" 하고 해석합니다.
* **터미널 셸:** `git commit -m "..."`이라고 치면, 셸은 `\`와 `n`이라는 두 개의 글자가 들어온 것으로 보고 그대로 Git에게 전달합니다. Git은 전달받은 글자 그대로를 커밋 메시지에 저장해 버리는 거죠.

---

## 2. 셸마다 해석 방식이 달라요
사실 `\n`을 해석하게 할 수는 있지만, 사용하는 셸(Bash, Zsh, PowerShell 등)에 따라 문법이 다 제각각입니다.

* **Bash / Zsh (리눅스/맥):** 앞에 `$`를 붙여야 특수 문자로 해석합니다.
  ```bash
  git commit -m $'feat: details\n- half design'
  ```
* **PowerShell (윈도우):** 억음 부호(Backtick, `` ` ``)를 사용해야 줄바꿈이 됩니다.
  ```powershell
  git commit -m "feat: details`n- half design"
  ```

이처럼 환경마다 다르기 때문에, Git은 **"차라리 `-m`을 여러 번 쓰거나 에디터를 띄우는 방식"**을 표준처럼 권장하게 된 것입니다.

---

## 3. 가장 확실하게 줄바꿈을 넣는 "정석"
명령어 뒤에 `-m`을 붙이지 않고 그냥 `git commit`만 쳐보세요.



그러면 설정된 기본 에디터(Vim이나 Notepad 등)가 열립니다. 거기서는 **메모장에 글을 쓰듯이** 엔터를 쳐서 줄바꿈을 마음껏 할 수 있고, 저장하고 닫으면 그 내용이 그대로 커밋 메시지가 됩니다. 이게 가장 실수가 적고 정석적인 방법이에요.

---

### 💡 요약하자면
`\n`이 안 먹히는 건 질문자님의 잘못이 아니라, **터미널이라는 환경이 `\`를 특수 기호로 처리하지 않도록 설계되었기 때문**입니다.

앞으로는 **`-m`을 문단마다 여러 번** 쓰시거나, 아예 **`git commit`만 쳐서 에디터를 활용**해 보세요. 훨씬 편하실 거예요!

**혹시 지금 윈도우(PowerShell) 환경인가요, 아니면 리눅스/맥 환경인가요?** 환경에 맞는 가장 편한 단축키나 설정법을 알려드릴 수 있습니다.
